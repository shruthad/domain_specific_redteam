We built a reusable utility for creating and evaluating domain-specific LLM red-team benchmark datasets, starting with banking and finance. The system first defines a structured finance risk taxonomy mapped to OWASP LLM risks, MITRE ATLAS-style attack coverage, NIST AI RMF functions, and finance-specific controls such as KYC, AML, lending, payments, customer data protection, fraud monitoring, and investment advice. From this taxonomy, it generates safe seed prompts for each risk category and converts them into validated benchmark records with fields such as attack_id, risk_category, attack_type, difficulty, expected_behavior, unsafe_success_criteria, safe_response_guidance, and attack_query.

The utility then expands and strengthens the dataset using optional red-team tooling. DeepTeam is used as an attack-expansion layer to create adversarial variations such as roleplay, policy bypass, prompt injection, RAG exfiltration, data extraction, and tool misuse scenarios. Garak is used as a scanner-style coverage layer to add common LLM vulnerability patterns such as hallucination, jailbreaks, data leakage, prompt injection, and encoding/obfuscation attacks. After generation and expansion, the pipeline normalizes records, removes duplicates, validates schema and safety rules, and exports the final benchmark as finance_redteam_attacks.jsonl. It also creates a Promptfoo-compatible YAML file so the same dataset can be evaluated against a target model such as Gemini.

The project is designed so other users can run it end-to-end through notebooks without changing the core code. Notebook 01 creates and expands the dataset from the selected YAML config, while Notebook 02 evaluates the generated attacks against Gemini using Promptfoo. Users can start with the default deterministic config, which requires no paid API, or enable the Gemini-backed DeepTeam config when they want live LLM-generated attack variations. To adapt the utility for another domain, teams can replace the taxonomy, seed prompts, mappings, expected behavior, and unsafe success criteria while keeping the same pipeline: domain risks → safe prompts → optional expansion → validation → JSONL export → model evaluation. This makes the project a reusable foundation for building governed, auditable red-team datasets across different domains.
